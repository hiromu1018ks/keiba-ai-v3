Phase 3 の leak-safety backbone は **populated columns について堅牢** に構築されている。行レベル §13.2 stamp は完備、cutoff 算術は `race_date - 1 day` で正確、rolling は strict `<` + per-observation `obs_id` window で CYCLE-2 HIGH #1 cross-observation leak を回避、allowlist は fail-loud で禁止 timing ゼロ、Parquet は byte-reproducible で SHA256 一致を再検証済。これらはPhase 1 D-01/D-13/D-14・CLAUDE.md §13.2/§14.3 契約を遵守している。

しかし GOAL「every feature row is point-in-time correct」の含意である **feature matrix の完結性**について、1つの実質的 gap が存在する：

1. **CR-01（ silent empty features）**: rolling_babacd_* / rolling_timediff_* の6 registered feature column が PyArrow probe で 100% NaN であることを実証した。これは leak ではなく silent contract breach だが、CORE VALUE「リーク防止**と再現性**」の再現性側を損なう。manifest は `feature_count: 63` を謳い、registry はこれらを実 feature として登録し、unit test は合成 dict fixture でのみ検査するため end-to-end で検出されない。Phase 4 は宣言より6列少ない rolling features で学習することになる。構造的 leak ではないため BLOCKER ではなく PARTIAL だが、OWNERS が「リーク防止と再現性は聖域」と明言した以上、registry-実体乖離を残したまま PASS とは呼べない。

加えて4つの Critical/Warning が潜伏負債として残存（CR-02 unfiltered JOIN・CR-03 silent fallback・CR-04 pickle load・WR-01 running_style leak）。これらは現時点で must-have を直接破るものではないが、Phase 4/8 で顕在化するリスクを抱える。特に WR-01 は小規模だが real look-ahead leak であり、registry が宣言する cutoff rule と実装が矛盾している。

**推奨 next-step**: gap closure plan で (a) CR-01 の6エントリを registry/rolling.py から削除するか source を供給する、(b) WR-01 に PIT pre-filter を追加する。これらは小規模（< 1 plan）で閉じる。CR-02/03/04 は Phase 8 対抗監査または別 quick fix で対処可能。
