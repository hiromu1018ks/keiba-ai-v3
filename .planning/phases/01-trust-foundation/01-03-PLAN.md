- **HIGH #5:** ETL を **idempotent** にする。`CREATE TABLE IF NOT EXISTS` + 素の `INSERT` では再実行で行が重複する（§19.1 再現性違反）。**staging-table-swap**（`_create_normalized_tables` で `normalized.n_race_tmp` を作成 → INSERT → `DROP normalized.n_race; ALTER TABLE n_race_tmp RENAME TO n_race` を1トランザクションで実行）または `INSERT ... ON CONFLICT (year, jyocd, kaiji, nichiji, racenum) DO UPDATE SET ...` のいずれかを採用。`test_etl_idempotent_rerun` が「2回 ETL を実行しても件数とハッシュが同一」ことを assert する。
- **HIGH #6:** ETL 書込接続は plan 01-01 Task 2/3 で定義した **ETL ロール（`KEIBA_ETL_DB_USER` / `Settings.etl_dsn` / `write_pool` / `write_cur` fixture）**を使用する。plan 01-03 単独で normalized 書込ロールを定義しない。
- **HIGH #8:** DB-test skip policy を **fail-by-default unless `KEIBA_SKIP_DB_TESTS=1`** に統一（plan 01-01/01-02 と同一・`.env` 未設定時 skip は廃止）。
- **MEDIUM (#1 n_uma_race under-specified):** `normalized.n_uma_race` の型付きカラム定義を `n_race` と同等に詳細化する（PK: year/jyocd/kaiji/nichiji/racenum/umaban/kettonum、kakuteijyuni integer、bamei text、kisyu text、touzai varchar(2)、norei integer、zogenfuka real 等・docs/everydb2/04-UMA_RACE.md 等を参照）。
- **MEDIUM (#2 pg_stat unreliable as sole proof):** `pg_stat_user_tables.n_tup_upd/n_tup_del` を「補助シグナル」に格上げし、**主証明は row-hash + row-count + INFORMATION_SCHEMA 権限テスト**にする。`n_tup_upd/n_tup_del` は VACUUM でリセットされるため単独では不十分。
- **MEDIUM (#3 date.fromisoconfig typo):** `load_class_config` で `date.fromisoformat(...)` を使用（`fromisoconfig` ではない）。

5. **HIGH #5 直接実装（idempotent staging-swap）:** `def _idempotent_load(write_cur, table: str, rows: list[tuple], columns: list[str]) -> int`:
   - ` staging = f"{table}_staging"`
   - `write_cur.execute(f"CREATE TABLE IF NOT EXISTS normalized.{table} (LIKE normalized.{table} INCLUDING ALL)")`（初回のみ・テーブルが無ければ作成）
   - `write_cur.execute(f"CREATE TABLE IF NOT EXISTS normalized.{staging} (LIKE normalized.{table} INCLUDING ALL)")`
   - `write_cur.execute(f"TRUNCATE normalized.{staging}")`
   - `write_cur.executemany(f"INSERT INTO normalized.{staging} ({','.join(columns)}) VALUES ({','.join(['%s']*len(columns))})", rows)`
   - `write_cur.execute(f"DROP TABLE IF EXISTS normalized.{table}")`, `write_cur.execute(f"ALTER TABLE normalized.{staging} RENAME TO {table}")` — これらは呼出元の transaction 内で実行され atomic に commit される
