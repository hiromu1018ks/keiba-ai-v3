## Design Decisions

### D-09.1-01: 必須 5 特徴量は `_SPEED_FIGURE_AXES` 拡張（D-09 踏襲）

`rolling.py::_SPEED_FIGURE_AXES` タプルに (axis, window) ペアを追加し、既存の speed_figure
専用集約ブロック（`if system == "speed_figure":`）を拡張。sentinel/count ルールは D-09 と同一:

- `count >= window` で算出、未満は `__MISSING__`（D-11）
- `count` 軸は常に実際 count（0〜window）を出力（D-11 信頼度軸）

各 axis の集約:
- `median`: `windowed.groupby("obs_id")["_sf_value"].median()`（mean と同構造）
- `best2_mean`: obs_id 毎に `_sf_value` の上位2件（`nlargest(2)`）の平均。**count >= 2 で算出**
  （上位2件が必要・window=5 窓内）。count<2 は `__MISSING__`
- `trend_last_minus_mean5`: window=5 窓で `first(直近1件) − mean(5件)`。count >= 5 で算出
- `trend_mean3_minus_mean5`: window=5 窓で `head(3).mean() − mean(5件)`。count >= 5 で算出

### D-09.1-02: same_surface / same_distance_bucket は別経路（条件 match filter）

`_SPEED_FIGURE_AXES` でなく、speed_figure 集約ブロック内の別サブブロックで処理:
`recent`（PIT filter + `groupby("obs_id").head(5)` 済み・LOOKBACK=5 窓）上で条件 match する行のみで集約。

- **target surface**: `observations["trackcd"]` → `_derive_surface`（`speed_figure.py` から import・
  過去走 `history["surface"]` と同一関数で一貫性保証）。obs_id → target_surface mapping を構築
- **target distance_bucket**: `observations["kyori"]` → bucket 分類（D-09.1-04）。obs_id → mapping 構築
- **過去走 surface**: `history["surface"]`（`compute_speed_figure_for_history` が既に付与済み）
- **過去走 distance_bucket**: `history["kyori"]` → bucket 分類
- `recent` の各行に obs_id で target 条件を付加し、過去走条件 == target 条件の行で mean/max/count

PIT 保証: `recent` は既に `as_of_datetime < feature_cutoff_datetime`（strict `<`）で filter 済み。
その部分集合（条件 match）も PIT-correct。target race 当日結果は混入しない。

### D-09.1-03: same_surface / same_distance_bucket の sentinel ルール（D-09 から意図的逸脱・文書化）

D-09 の `count >= window(5)` を same_surface/distance_bucket には適用しない:

- **mean/max**: `count >= 1`（同条件1走以上で算出）。理由: 直近5走窓で同条件走は通常 1-3 走。
  `count >= 5` だと大半が `__MISSING__` となり feature が実質無効化される。同条件1走でも
  「その条件での1回の走破」は field_strength 入力として有用な情報
- **count_5**: 直近5走窓での同条件走数（実際 count 0-5）を常に出力（D-11 信頼度軸踏襲）。
  モデルが少サンプルの mean/max の信頼度を判断する材料

この逸脱は `feature_availability.yaml` および本 PLAN で文書化し、unit test で機械保証する。

### D-09.1-04: distance bucket 定義（conservative 固定 bucket）

`speed_figure.py` に `derive_distance_bucket(kyori)` 関数を追加（`_derive_surface` と対称・
rolling と speed_figure で共利用）。JRA 平地主距離を標準的な sprint/mile/classic/staying に分類:

| bucket | kyori 範囲 | JRA 主距離 |
|--------|-----------|-----------|
| short | < 1400 | 1000, 1200 |
| mile | 1400-1799 | 1400, 1500, 1600, 1700 |
| middle | 1800-2199 | 1800, 2000 |
| long | >= 2200 | 2200, 2400, 2500, 2600, 3000, 3200 |

障害（`surface == 'obstacle'`・trackcd 51-59）は `speed_figure` で既に NaN 除外済み・
distance_bucket 対象外。NaN/未知 kyori は `__MISSING__` bucket（同条件 match しない→count 増えない）。

### D-09.1-05: target trackcd/kyori の observations 追加と feature_matrix 非出力

`builder.py::_OBS_DB_SELECT_COLUMNS` に `nr.trackcd AS trackcd`, `nr.kyori AS kyori` を追加
（target race の条件・D-09.1-02 の same_surface/distance_bucket 計算に必要）。

- **feature_matrix には出力しない**: `trackcd`/`kyori` は `rolling_df` から `rolling_*` 列のみ
  merge する仕組み（builder L524-540）により rolling 結果には入らない。しかし `_fetch_feature_sources`
  が返す `obs_df`（= feature_matrix 本体）には残るため、Step 6b で `obs_id` と共に明示 drop
  （`feature_matrix.drop(columns=["trackcd", "kyori"], errors="ignore")`）
- **registry parity**: `trackcd`/`kyori` は registry に race_context (both_allowed) として既登録だが、
  v1.0/v1.1 でも feature_matrix に出力されていない（registry は利用可能 feature の catalog・
  出力必須でない）。本 Phase でも feature としては出力せず（rolling 計算の中間値のみ）。
  `assert_matrix_columns_registered` は「出力した feature が全て registry にある」方向の検査なので、
  drop すれば parity は維持（出力しない feature があっても違反でない）

### D-09.1-06: schema_version bump 0.4.0 → 0.5.0

`feature_availability.yaml` の `schema_version` を 0.4.0 → 0.5.0 に bump（11 feature 追加・
§19.1 再現性）。新 snapshot の `feature_availability_version` = "0.5.0"。
