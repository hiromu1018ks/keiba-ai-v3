### WR-01: best2_mean vector化書き換えが `sort_values(kind=)` 未指定で非決定論的リスク

**Files modified:** `src/features/rolling.py`
**Commit:** `fc78cd2`
**Applied fix:** `src/features/rolling.py:464-487` の `best2_mean` 軸 vector 化処理で
`sort_values(["obs_id", "_sf_value"], ascending=[True, False])` に `kind="mergesort"`
（安定ソート）を追加。tie-break が入力順序に従い決定論的になり、byte reproducibility
を「偶然 mean が同値」でなく「sort 安定性」で担保する。`split_3way` idiom と整合。

**検証結果:**
- Tier 1: 修正箇所再読で修正内容・周辺コードの完全性を確認
- Tier 2: `python3 -c "import ast; ast.parse(...)"` で構文 OK
- Tier 3 (unit test): `test_speed_figure.py` / `test_speed_figure_pit.py` /
  `test_speed_figure_builder_integration.py` / `test_builder.py` が 38 passed
  （pre-existing の snapshot 欠落テスト 2 件を deselect）

### WR-02: `_compute_pit_par` Stage 3 が row-by-row `out.at[idx, ...]` で決定論性を損なう可能性

**Files modified:** `src/features/speed_figure.py`
**Commit:** `352d4d8`
**Applied fix:** `_compute_pit_par` Stage 3 (obs_id 単位 all-history median) を
row-by-row `out.at[idx, ...]` から vectorized reindex に統一。Stage 1/2 と対称化。
同じ idiom を `_compute_leave_one_out_variant` Stage 2/3 にも適用し、
`median2.get + notna 三項` → `reindex(...).fillna(0.0)` で対称化。

**等価性の検証（慎重検証）:**
- `grp3` は pending 行の obs_id を group key に持つため `reindex` で NaN にならない
- `size3.get(obs_id, 0)` → `reindex(...).fillna(0).astype(int)` で等価
- `median3.get(obs_id, NaN)` → `reindex(...)` で等価

**検証結果:**
- Tier 1: 修正箇所再読で確認
- Tier 2: 構文チェック OK
- Tier 3 (unit test): 38 passed・PIT guard adversarial test
  (`test_speed_figure_pit.py`) 含む全 GREEN・値の変化なし

**ステータス:** `fixed`（ただし logic 等価性の検証なので人間確認推奨）

### WR-03: `_compute_pit_par` Stage 2 で positional alignment の暗黙前提リスク

**Files modified:** `src/features/speed_figure.py`
**Commit:** `a020c70`
**Applied fix:** `_compute_pit_par` と `_compute_leave_one_out_variant` の両関数で
`out = expanded_*.copy()` の直後に `out = out.reset_index(drop=True)` を追加。
obs_id expand で同一元 history 行が複数 observation に複製されて duplicate index が
発生するが、duplicate index では `out.index[pending]` と `out.loc[pending].index` が
異なる順序を返す pandas 既知の落とし穴（positional alignment の暗黙前提崩れ）を
構造保証する。

**検証結果:**
- Tier 1: 修正箇所再読で確認
- Tier 2: 構文チェック OK
- Tier 3 (unit test): 38 passed・回帰なし

## snapshot SHA256 への影響（重要・必須記録事項）

**WR-01（sort kind）と WR-03（reset_index）は、実データに tie（同一 speed_figure 値）
や duplicate index が存在する場合、出力順序が変わり snapshot の SHA256 が変わりうる。**
これは byte reproducibility の聖域に直結するため、以下を明記する:

- 修正後、unit test の `test_byte_reproducible_speed_figure`（合成データ）が GREEN でも、
  実 snapshot（`snapshots/feature_matrix_20260626-1a-speedprofile-v1`）の SHA256 が
  変わる可能性がある
- snapshot 再生成（~13.5分・live-DB）は orchestrator が別途判断するため、fixer は実施しない
- **WR-01/WR-03 適用後は snapshot 再生成で SHA256 再検証が必要**
  （manifest.yaml の SHA256 hash 値との照合）

WR-02 は Stage 3 fallback の row-by-row を vectorized reindex に等価変換したもので、
値の変化はない（unit test 38 passed で確認）。SHA256 影響は論理的に無いが、
float 演算順序の変化で丸め誤差が微細に変わる可能性は完全には否定できないため、
snapshot 再生成時の再検証対象に含める。
