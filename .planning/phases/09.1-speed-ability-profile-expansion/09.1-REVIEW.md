### WR-01: best2_mean vector化書き換えが `sort_values(kind=)` 未指定で非決定論的リスク

**File:** `src/features/rolling.py:470-476`
**Issue:**
`best2_mean` 軸の vector 化 (5倍高速化) で `apply + nlargest` から `sort_values + groupby.head(2) + mean` に置換した。しかし `sort_values(["obs_id", "_sf_value"], ascending=[True, False])` が `kind=` を指定せず、pandas の default quicksort は不安定ソート。同一 `_sf_value` の tie が2件以上ある場合、`groupby.head(2)` が選ぶ2件の順序が非決定論的になる。

現状は `mean()` が線形和のため、選ばれる2件の値が同じなら mean も同値 (bit-identical 維持)。これは `test_byte_reproducible_speed_figure` が GREEN な理由。しかし byte reproducibility の聖域は「アルゴリズムが決定論的」で担保すべきで、「偶然 mean が同値」に依存すべきでない。将来 `best3_mean` / `count` 系に拡張した際、同じ idiom が非決定論性を manifest する温床になる。

**Fix:**
```python
top2 = (
    valid_windowed.sort_values(
        ["obs_id", "_sf_value"], ascending=[True, False], kind="mergesort"
    )
    .groupby("obs_id", sort=False).head(2)
)
```
`kind="mergesort"` (安定ソート) を指定することで、tie-break が入力順序に従い決定論的になる (split_3way の idiom と整合)。byte reproducibility を「mean の偶然同値」でなく「sort 安定性」で保証する。

### WR-02: `_compute_pit_par` Stage 3 が row-by-row `out.at[idx, ...]` で決定論性を損なう可能性

**File:** `src/features/speed_figure.py:425-437`
**Issue:**
`_compute_pit_par` の Stage 3 fallback (obs_id 単位の all-history median) が `for idx in out.index[pending]:` で1行ずつ `out.at[idx, ...]` を設定する。本ループは `median3.get(obs_id)` を使うため値自体は決定論的だが、`out.index[pending]` の順序が groupby 操作後の `out` の index 順序に依存する。input フレームの row 順序が非決定論的に変化すると (例: 今後の最適化で sort が入る)、`out.at[idx, ...]` 設定順序は変わるが値自体は正しい。

直接のバグではないが、Stage 1/2 (vectorized reindex) と Stage 3 (row-by-row) の混在がコードの決定論性監査を困難にする。Stage 3 も vectorized reindex に統一すべき。

**Fix:**
```python
# Stage 3: vectorized reindex (Stage 1/2 と対称)
pending = out["par_sec"].isna()
if pending.any():
    grp3 = out.loc[pending].groupby("obs_id", sort=False)["time_sec"]
    size3 = grp3.size()
    median3 = grp3.median()
    idx3 = out.loc[pending].set_index(["obs_id"]).index
    size3_map = size3.reindex(idx3)
    median3_map = median3.reindex(idx3)
    pending_indices = out.index[pending]
    out.loc[pending_indices, "par_sec"] = median3_map.values
    out.loc[pending_indices, "sample_count"] = size3_map.values.astype(int)
    out.loc[pending_indices, "fallback_level"] = "all_day"
```
これで Stage 1/2/3 全てが vectorized reindex に統一され、byte reproducibility 監査が容易になる (現状の `test_byte_reproducible_speed_figure` GREEN は維持)。

### WR-03: `_compute_pit_par` Stage 2 で `size2_map[mask2]` を `selected` に設定する際の位置依存リスク

**File:** `src/features/speed_figure.py:416-423`
**Issue:**
```python

size2_map = size2.reindex(idx2)
median2_map = median2.reindex(idx2)

```
`mask2` は `size2_map` (idx2 に reindex) の `.values` 配列。`pending_indices = out.index[pending]` は `out.loc[pending]` と同じ順序、`idx2 = out.loc[pending].set_index(key2).index` も同じ行順序を保持するため、`mask2` と `pending_indices` は位置的に整合する。本検証で実際に GREEN になることを確認した。

**Fix:**
`out = expanded_filtered.copy()` の直後に `out = out.reset_index(drop=True)` を追加し、index を一意化する。これで `out.index[pending]` と `out.loc[pending].index` が常に同一順序を返す (positional alignment の暗黙前提を構造保証)。同じ idiom は `_compute_leave_one_out_variant` にも適用すべき。

### WR-04: `same_surface` / `same_distance_bucket` の sentinel が count>=1 で少サンプル mean のノイズ源になる

**File:** `src/features/rolling.py:532-603`, `src/config/feature_availability.yaml:421-468`

### WR-05: `orchestrator._calibrate_catboost_manual` が strict-later disjoint guard を `race_date.min/max` で検証し、同日複数レースで弱くなる

**File:** `src/model/orchestrator.py:643-651`
**Issue:**
```python
calib_min_date = calib_df.loc[X_calib.index, "race_date"].min()
train_max_date = train_df["race_date"].max()
if not (train_max_date < calib_min_date):
    raise ValueError(...)
```
guard は「train_max_date < calib_min_date」を検証する。これは `split_3way` が完全時系列条件 (train_max < calib_min < calib_max < test_min) を保証するため redundant だが、`split_periods` 指定 (Phase 5 D-03 BT窓) の場合、`split_3way` の guard と本 guard の2層になる。

問題は `calib_min_date` が「calib の race_date の最小」である点。同一 race_date に複数 race がある場合、train 側の同 race_date の race と calib 側の同 race_date の race が競合する。`split_3way` は race_date の `between` で分割するため、train_end == calib_start の同一 date の race が両側に分配される。これは review MEDIUM#5 の guard では `train_max < calib_min` が成立するが、実質的に同一日付の race が train/calib 両方に入る look-ahead leak のリスクがある (同日午前のレース結果が同日午後のレース予測に漏れる)。

`split_3way` の guard は `train_max < calib_min` を要求するため、train_end < calib_start は保証される。したがって同日配分は生じない (race_date 単位で完全分離)。本 guard は `split_3way` の guard と等価で冗長だが、`calibrate_model` (LightGBM パス) と対称にする意図は理解できる。

ただし `_calibrate_catboost_manual` の guard は `calib_min_date` が `X_calib.index` に限定されている点に注意。`X_calib` は `make_X_y(calib_df)` の出力で、`calib_df` の全行が含まれるはず (filter_eligible で除外されない限り)。もし `X_calib` が空 (calib 全行が non-eligible) の場合、`.min()` が NaT になり、`train_max_date < NaT` は False になり guard が raise する (正しい fail-loud)。問題なし。

**Fix:**
冗長だが harmless。Phase 11 (calibration 精査) で `split_3way` の guard に一本化するか、hour 単位の厳密な時刻分離 (race_start_datetime) に移行すべきか検討記録する。

### WR-06: `feature_availability.yaml` が `trackcd` / `kyori` feature を registry 登録するが builder が Step 6b で drop する silent parity drift

**File:** `src/config/feature_availability.yaml:119-132`, `src/features/builder.py:625-627`
**Issue:**
registry は `trackcd` と `kyori` を race_context feature (`both_allowed`) として登録する (L119-132)。しかし builder は D-09.1-05 の設計判断で Step 6b で `trackcd/kyori` を `feature_matrix.drop(columns=["obs_id", "trackcd", "kyori"], errors="ignore")` で明示的に削除する。結果として registry 登録 feature が feature_matrix に出力されない。

`assert_matrix_columns_registered` (availability.py:314-348) は「出力カラムが allowlist (registry + reserved + ...) に含まれるか」を検査するが、「registry 登録 feature が全て出力されるか」は検査しない。したがって registry↔実体 silent parity drift が検出できない。

`_derive_feature_columns` (data.py:179-211) が `(registry 実在 feature) & (Parquet 実カラム)` の積を取るため、Parquet に存在しない `trackcd/kyori` は FEATURE_COLUMNS に自動的に含まれない。これは「silent drift を mask する」方向に働き、registry が `trackcd/kyori` を feature として宣言しているのに model が消費しない状態を許容する。

**Fix:**
以下のいずれかで一貫性を回復すべき:
1. registry から `trackcd/kyori` feature を削除 (current 実体に合わせる)
2. builder Step 6b で `trackcd/kyori` を drop せず feature_matrix に出力する (registry に合わせる)

Phase 9.1 の D-09.1-05 設計判断 (speed profile 特徴量のみ追加・snapshot 構成変更を最小化) を尊重するなら (1)。ただし `trackcd/kyori` が model feature として有用な場合は (2) も検討すべき。現状は registry が「registry に登録すれば model が使う」契約を暗中に破っている。

```python
    if max(hit_rates) > min(hit_rates):
        signal_present = True
```
同一 market bucket 内で model_p 高低による的中率差が1つでも cell に残れば `signal_present=True`。これは非常に弱いシグナル判定で、ランダムノイズで容易に True になる。D-15 は「正式結論は出さない (Phase 12 EVAL-02 に委譲)」と明記するため harmless だが、stop gate verdict が residual_signal に依存する (D-16) ため、弱いシグナルで「Phase 10 進行候補」判定になりやすい。

**Fix:** Phase 12 EVAL-02 で p値判定 / 回帰係数に移行する前提で、現状は記録のみ。D-16 verdict が residual_signal だけで「進行候補」になるロジック (`_decide_stopgate_verdict`) は、Phase 10 進行前に人間が再確認すべき (honest 記録の intent は保たれている)。
