## Summary

Phase 3.1「Timediff/Babacd Rolling Restoration」の実装に対する deep review。対象は (A) rolling feature 6個（timediff/babacd 系統）の復元、(B) advisory 4件（WR-01'/WR-02/CR-01新/WR-03）の hardening、3者 parity（registry ↔ rolling ↔ availability）の整合性。

全体として advisory hardening は概ね機能している。WR-01'/WR-02 の fail-loud、CR-01新 persist→manifest 順序、CR-01 canonical key merge は実装されており、対応する unit test も配置されている。3者 parity（`_ROLLING_SYSTEMS` 3定義の一致）は `test_registry_rolling_systems_match_rolling_impl` で機械保護されている。

ただし以下の点が重大な懸念として残る:

1. **babacd 派生ロジックが `'0'` を信号欠損扱いして `__MISSING__` 化している** (CR-01)。JRA 標準の馬場状態コードでは `'0'` は「良馬場」の有効値である可能性が高く、本実装は babacd feature を過半数のレース（良馬場）で欠損化する data loss リスクを負っている。コード体系が EveryDB2 側で不完全（`CODE.md#2010` は Phase 1 D-10 で「スクレイピング不完全」と明記）なため確定できないが、実データ検証を行わずに本番 snapshot 生成に用いるのは危険。
2. **`_fetch_history` の DB例外時 silent empty フォールバックに対する fail-loud が無い** (WR-01)。obs 側は WR-02 で検知するが、history 側が DB例外で空 DF を返すと rolling/sentinel が「全馬 新馬扱い」で silent に進行する。leak ではないが silent data loss 経路として残っている。
3. **`test_no_registered_feature_column_all_nan_end_to_end` の fixture が非現実的な babacd 値 (`"1"`) を使い、CR-01 の babacd '0' 欠損バグを素通りさせている** (WR-03)。regression guard としての強度が不足。
4. **timediff parse が raw フォーマット `sNNN`（符号1桁+数字3桁）を前提とするが桁数無検証** (IN-01)。`"1000"` 等の異常フォーマットが 0.0 秒として混入する可能性がある。

詳細は各セクションを参照のこと。

## Critical Issues

### CR-01: babacd 派生が `'0'`（良馬場）を信号欠損として `__MISSING__` 化する data loss リスク

**File:** `src/features/builder.py:280-285`
**Issue:**
`_construct_derived_columns` の babacd 派生ロジックにおいて、芝・ダート双方の馬場状態コード `'0'` を信号欠損扱いして `__MISSING__` sentinel 化している:

```python
turf_valid = turf_mask & turf_val.notna() & (turf_val != "") & (turf_val != "nan") & (turf_val != "0")
babacd = babacd.where(~turf_valid, turf_val)
# ダート側も同様
dirt_valid = dirt_mask & dirt_val.notna() & ... & (dirt_val != "0")
babacd = babacd.where(~dirt_valid, dirt_val)
```

JRA 標準の馬場状態コード（`docs/everydb2/CODE.md#2010` 参照先の体系）では `'0'` は「良（fast）」を表す有効値である可能性が高い。`docs/everydb2/03-RACE.md` L76-77 に `SibaBabaCD varchar(1) 初期値 0` とあり、初期値 `'0'` は JRA 競馬場の良馬場設定時の既定値であるのが通例。もし `'0'` が良馬場であれば:

- 過半数の過去走レース（良馬場）が全て `__MISSING__` 化され、`rolling_babacd_mean_5` / `rolling_babacd_latest_5` / `rolling_babacd_sd_5` が「良馬場のレース」で一律 sentinel 扱いになる。
- 競馬馬場状態の推移分析（良→稍重→重 の騎手/馬の適性）が feature から消失し、Phase 4 モデルの予測力を構造的に損なう。
- §13.5「過去走馬場状態」feature 要求に対する実質的未実装状態になる。

コード体系自体が `CODE.md#2010` では「1=未設定初期値」しか記載されず不完全（Phase 1 D-10 で言及）のため、本 review では確定できない。しかし JRA 一般慣行と矛盾する可能性が高く、実データ検証を行わずに snapshot 生成・学習 dataset 構築に用いるのは危険。

**影響:**
- babacd 系 feature 3つ（mean/latest/sd）が良馬場レースで全欠損。
- `test_no_registered_feature_column_all_nan_end_to_end` は `"hist_sibababacd": "1"` の非現実的 fixture で通過するため silent regression が guard されない（WR-03 参照）。
- UAT で「良馬場データが多い時期」の snapshot を再生成すると SHA256 drift と見紛う原因にもなる。

**Fix:**
実データ検証が必須。検証結果により2通り:

(a) `'0'` が良馬場を意味する場合（高確率）:
```python
# '0' は良馬場の有効値なので欠損扱いしない
turf_valid = turf_mask & turf_val.notna() & (turf_val != "") & (turf_val != "nan")
babacd = babacd.where(~turf_valid, turf_val)
dirt_valid = dirt_mask & dirt_val.notna() & (dirt_val != "") & (dirt_val != "nan")
babacd = babacd.where(~dirt_valid, dirt_val)
```

(b) `'0'` が実際に未設定/初期値を意味する場合（要 EveryDB2 マニュアル確認）:
現状維持。ただしその場合、`docs/everydb2/CODE.md#2010` への実測補完と、`label_spec.yaml` への `babacd_sentinels` 明記（`timediff_sentinels` L133 と同形式）を行い、`_TIMEDIFF_SENTINELS` と対称な定数化を推奨。

いずれの場合も、live-DB で `SELECT sibababacd, count(*) FROM normalized.n_race WHERE sibababacd IS NOT NULL GROUP BY 1 ORDER BY 1` を実行し分布と意味を確定すること。

### WR-01: `_fetch_history` の DB例外 silent empty フォールバックに対する fail-loud が欠落

**File:** `src/features/builder.py:628-662`
**Issue:**
WR-02 は `_fetch_feature_sources`（obs 側）が空を返した場合 `build_feature_matrix` L398-402 で `RuntimeError` fail-loud する。しかし `_fetch_history`（history 側）は L653-662 で接続例外時に空 DF を返し、呼出側での空検知が無い。後続の Step 5 は `if len(history) > 0 and len(feature_matrix) > 0:` で rolling をスキップ、Step 6 も同様にスキップされる。結果として:

- DB 一時障害・接続例外で history だけ空になった場合、全馬が「新馬」扱い（`rolling_*_mean_5` 等が全て `__MISSING__`）の feature_matrix が生成される。
- leak ではないが、silent data loss として「障害時に弱体化した feature_matrix を完成扱いで manifest 書出まで進む」経路が残る。
- WR-02 advisory の intent「silent data loss 回避」が obs 側のみで部分的にしか達成されていない。

**Fix:**
`build_feature_matrix` の Step 5 直前（または WR-02 チョークポイント直後）に history 空検知を追加:

```python
if len(feature_matrix) > 0 and len(history) == 0:
    # 過去走 history が空（DB例外/0行結果）は silent data loss になるため fail-loud。
    # 新馬のみの実データという稀なケースは存在しない（JRA 全期間に過去走0の馬は稀）。
    raise RuntimeError(
        "history fetch が空結果を返した・DB例外/0行結果の silent empty を検知 "
        "(WR-01 fail-loud・advisory hardening・全馬新馬扱いの silent data loss 回避)"
    )
```

ただし「本当に新馬のみの snapshot を生成したい」ユースケースがある場合は、明示的な opt-in flag（例: `allow_empty_history=True`）で区別すること。

### WR-02: `test_no_registered_feature_column_all_nan_end_to_end` の fixture が非現実的な babacd 値を使い CR-01 を素通り

**File:** `tests/features/test_builder.py:155-158, 373, 477`
**Issue:**
CR-01 regression guard テストの fixture が `"hist_sibababacd": "1"`（稍重相当）を使っている。JRA データでは `'0'`（良）が多数派である可能性が高い。CR-01 の `'0'` -> `__MISSING__` 派生ロジックは `'1'` では発火しないため、この regression guard は CR-01 の data loss を検知できない。

```python
"hist_sibababacd": "1",              # 芝・良馬場（trackcd 第1桁1→芝→babacd=1）
```

コメントに「芝・良馬場（...babacd=1）」とあるが、JRA 標準では良馬場は `'0'` のはずで、fixture とコメントの整合性も崩れている。

**Fix:**
fixture を現実的な値の分布に近づける。最低でも `'0'` を含める:

```python
# 良馬場 '0' と稍重 '1' が混在する現実的な分布
"hist_sibababacd": "0" if day_offset == -2 else "1",
```

これにより CR-01 の `'0'` -> `__MISSING__` 派生がテスト入力に現れ、guard が実効性を持つ。

### WR-03: timediff parse の桁数・フォーマット無検証（異常フォーマットが 0.0 秒として混入）

**File:** `src/features/builder.py:252-258`
**Issue:**
timediff parse が raw フォーマット `sNNN`（符号1文字+数字3文字）を前提するが、入力の桁数・先頭文字の有効性を検証しない:

```python
raw_td = result["timediff_raw"].astype(str)
is_sentinel = raw_td.isin(_TIMEDIFF_SENTINELS)   # {"0000", "9999"}
sign = raw_td.str[0]
digits = pd.to_numeric(raw_td.str.slice(1, 4), errors="coerce")
signed = digits.where(sign != "-", -digits)
result["timediff"] = (signed / 10.0).where(~is_sentinel, np.nan)
```

- `"1000"` 等の先頭が数字の異常フォーマット: sign="1", digits=0, signed=0, /10=0.0。0.0 秒（同着の勝馬差）として誤って混入する。
- `"+"` 等の短い文字列: digits=NaN, signed=NaN, /10=NaN。silent NaN になる（一応安全だが sentinel 扱いでなく純粋 NaN 扱い）。
- `" +001"`（前後に空白）: `str.slice(1,4)` で `" +0"` → pd.to_numeric で NaN。silent NaN。

`errors="coerce"` が異常値を全て NaN にするため、異常フォーマットの大部分は NaN 化されるが、4桁数字のみで先頭が `1-9` のケース（例: `"1000"`, `"5001"` 等）は 0.0 秒として混入する。実データでこのパターンが存在するか未検証。

**Fix:**
フォーマット validator を追加:

```python
# フォーマット: 先頭[+-]、続く3文字が数字。それ以外は NaN。
format_ok = raw_td.str.match(r"^[+-]\d{3}$", na=False)
digits = pd.to_numeric(raw_td.str.slice(1, 4), errors="coerce")
signed = digits.where(sign != "-", -digits)
result["timediff"] = (signed / 10.0).where(~is_sentinel & format_ok, np.nan)
```

### WR-04: `_fetch_feature_sources` の except 節が接続例外以外のケースを silent empty にしない設計だが、ProgrammingError 系（テーブル未存在）は正しく re-raise されるか実行時確認が難しい

**File:** `src/features/builder.py:616-625`
**Issue:**
コメント「想定外例外（MemoryError / ProgrammingError / RuntimeError 等）は re-raise」とあるが、psycopg3 の `UndefinedTableError` は `ProgrammingError` のサブクラスであり、本 except には入らない。これは意図通り。ただし、`psycopg.errors.Error` 基底（全 psycopg 例外の親）を catch していないため、予期せぬ psycopg 例外カテゴリが将来追加された場合に WR-02 fail-loud を素通りする可能性がある。

例えば `psycopg.errors.AdminShutdown`（管理者による DB シャットダウン）は `OperationalError` でなく `AdminShutdown` 直系で、現 except では catch されず re-raise される。これは正しい挙動だが、psycopg3 の例外階層の変動（将来的に新カテゴリ追加）に対する防御が薄い。

**Fix:**
明示的に「接続系例外のみ catch、それ以外は re-raise」を強化する場合、psycopg3 の `errors.Connection` 系基底クラス（`psycopg.errors.Connection` は存在しないが、`OperationalError` + `InterfaceError` + `ConnectionError` で概ねカバー）を使う現状で十分。ただし、WR-02 advisory の intent を明確にするため、単体テストで `psycopg.errors.UndefinedTableError` を注入して re-raise されることを機械保証する test を追加推奨:

```python
def test_wr02_reraises_undefined_table_error(monkeypatch):
    """WR-02: ProgrammingError 系（UndefinedTableError）は re-raise される。"""
    import psycopg.errors
    builder = _get_builder()
    def raise_undefined(read_pool):
        raise psycopg.errors.UndefinedTableError("test")
    builder._fetch_feature_sources = raise_undefined  # モックではなく実経路
    # 実経路の差し替えが難しい場合は _fetch_feature_sources 内部の cursor.execute を patch
```

### WR-05: rolling の categorical ブロックが `recent[system].astype(object)` を使うが、jyocd が None/NaN 混在の場合 `notna()` の挙動が str dtype で非自明

**File:** `src/features/rolling.py:265-274`
**Issue:**
jyocd は history SELECT で `ur.jyocd AS jyocd` で取得。DB から NULL が来た場合、pandas は object dtype で None を入れる。`raw_series = recent[system].astype(object)` は None を保持し、`.notna()` は None を False で返す。これは正しい。

だが、`recent[system].astype(object)` 自体が既に object の場合 no-op で、もし将来 jyocd が Int64 等の数値 dtype に変更された場合（例えば ETL 側で `pd.to_numeric` が適用された場合）、astype(object) で `pd.NA` が `nan` になり、`.notna()` で False 扱いになる。現在は str 扱いなので問題ないが、catalogical 系統の source 列が数値に変更される refactor で silent 挙動変化のリスクがある。

**Fix:**
明示的に文字列化してから処理する、または categorical 系統の source dtype を invariant として明記:

```python
# categorical 系統の source は必ず str dtype であることを前提とする。
# 数値化 refactor が必要な場合は categorical 扱いを外す（_CATEGORICAL_SYSTEMS から削除）。
raw_series = recent[system].astype("string")  # nullable string dtype で NA を明示
```
