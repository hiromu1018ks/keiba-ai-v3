**Researched:** 2026-06-19
**Domain:** Phase 2 normalized ETL 拡張 + Phase 3 rolling 3者再登録 + advisory 4件 hardening（Python/PostgreSQL/pandas/LightGBM 前提の feature engineering）
**Confidence:** HIGH（全て実データ検証 + 既存コード精読 + EveryDB2 公式マニュアル照合で確定。`[ASSUMED]` 項目なし）

## Summary

Phase 03.1 は Phase 3 gap-closure (03-05) で silent-empty breach 解消のために一時削除した `rolling_timediff_*` / `rolling_babacd_*` 計6 feature を復元し、Phase 1-A rolling features を 18 → 24（8系統×3軸）に拡張する挿入フェーズである。CONTEXT.md が指摘した通り、本フェーズの真の主作業は「機械的カラム取り込み」ではなく**設計特徴量の派生実装**である。実データ検証の結果、以下の3点が確定した。

第一に、**timediff parse 仕様は実データで完全確定**した。`public.n_uma_race.timediff` (varchar(4)) は全554,267行が length=4・NULL=0件。符号分布は `+`509,756件 / `-`39,629件 / `9`(=sentinel)4,506件 / `0`(=edge case)376件。1着馬(kakuteijyuni='01')は全て `-` 符号（MIN=`-000`=同タイム2着・MAX=`-056`=5.6秒差・39種）、2着以下は全て `+` 符号（EveryDB2 04-UMA_RACE.md field 66 TimeDIFN 仕様通り）。**先頭`0`の376行は全て `0000`** で kakuteijyuni='00'（取消/除外/中止）に完全一致 → `0000` は sentinel。`9999`(4,506件)も kakuteijyuni='00' に完全一致 → sentinel。両者とも `timediff_sentinels` に追加して NaN 化する。値=`NNN/10`秒（符号付き）。第二に、**babacd 派生設計は trackcd 第1桁分岐で確定**した。`public.n_race` の `sibababacd`/`dirtbabacd`/`trackcd` は実在（全て varchar）。trackcd 実分布から第1桁 `1`/`2`=芝（sibababacd active・dirtbabacd='0'）、`5`=ダート/障害混合（dirtbabacd active または sibababacd active）。障害競走(syubetucd='18'/'19'=label_spec model_ineligibility で既に学習除外)を含む `5` 系は history に混入するが、派生不能時 `__MISSING__` sentinel で安全。babacd 序数コードは CODE.md §2010 が不完全（`0`のみ）のため、実データ値 `0,1,2,3,4`（0=良/未設定・1→4=重）を D-02 通り**生序数コード数値扱い**（jyocd 前例）で確定。第三に、**advisory 4件の file:line 修正点は todo 処方箝 + 現状コード精読で確定**した。WR-01'(`builder.py:353-354` silent fallback)、WR-02(`builder.py:455-457,485-487` except→空DF)、CR-01新(`run_feature_build.py:206-228` manifest→persist 順序)、WR-03(`rolling.py:236-240` groupby().apply)。

**Primary recommendation:** (1) `normalize.py` の n_uma_race に `timediff`(varchar) を1カラム追加 + n_race に `sibababacd`/`dirtbabacd`/`trackcd`(全varchar) を3カラム追加し、staging-swap idempotent ETL を再実行。(2) `builder.py` の history SELECT/派生ロジックで timediff を real parse（sentinel `0000`/`9999`→NaN）+ babacd を trackcd 第1桁分岐で派生（芝→sibababacd・ダ→dirtbabacd・不能→`__MISSING__`）。(3) rolling.py / availability.py / feature_availability.yaml の3者に timediff/babacd を再登録（6→8系統）。(4) advisory 4件を hardening。(5) テスト5関数を更新。(6) live snapshot rebuild + parity 再検証。

### Locked Decisions
- **D-01: babacd は完全復元（24 feature・8系統×3軸）** — `normalized.n_race` に `sibababacd`(芝馬場)/`dirtbabacd`(ダート馬場)/`trackcd`(芝ダ判別) の3カラムを追加し、history builder で各過去レース行の `trackcd` を見て対応する馬場カラムを選択し babacd を派生する。babacd は `days_since_prev`/`race_nkey` と同じ「派生 history カラム」として扱う。SC#2（rolling 18→24）を維持。ユーザー選択: A（完全復元）。馬場適性信号を Phase 4 モデルに入れる意図。
- **D-02: babacd 集約 = mean/latest/sd の3軸・生の序数コードを数値扱い（特別リマップなし）** — `jyocd`（競馬場コード=2桁序数コード）の既存 rolling 前例と一貫。GBDT は序数値を自然に処理するため mean/sd も意味を持つ。`babacd` code→序数の順序付けは researcher が `docs/everydb2/CODE.md#2010`（馬場状態コード・スクレイピング不完全・要補完）で検証する。特別リマップは将来の改善余地として D-08 に defer。
- **D-03: timediff は標準（parse + sentinel除外）** — 符号付き文字列 `[+-]NNN`（NNN=3桁・値=`NNN/10`秒・1着馬は負値・2着以下は正值）を real に parse。sentinel `9999`（取消/除外/中止）は rolling 集約から除外→NaN。取消/除外/中止の過去走は timediff に寄与せず、既存 rolling の `__MISSING__`/count 扱いと一貫。parse 書式の詳細（先頭`0`の1,028行 edge case 等）は researcher が実データで確定。ユーザー選択: 標準。
- **D-04: advisory 4件の hardening は todo 処方箝通り・Claude/計画者に委ねる** — 各 advisory の file:line + 修正方針 + テスト条件は `.planning/todos/pending/phase3-advisory-hardening.md` に固定済み。WR-02 の fail-loud 実装スタイルは「チョークポイント検出」を採用（`_fetch` は log+空DF返却のまま、`build_feature_matrix` が「空結果/全NaN」を単一検証点で fail-loud → DB例外・0行結果・全NaN の全 silent-empty 系を一括捕捉・メッセージ明確）。ユーザー選択: 任せる。
- **D-05: datakubun は normalized に取り込まない** — `normalized.n_uma_race` は既に 554,267 行に dedupe 済み（version選別はETL側完結）のため、timediff 取得・babacd 派生のいずれにも不要。roadmap SC#1（timediff/babacd のみ）を厳格に維持。将来の細かい history フィルタ用途（障害戦除外等）は D-08 に defer。ユーザー選択: 取り込まない。

### Pattern 1: 派生 history カラム（DB 非実在・pandas で構築）
**What:** DB に実在しないが rolling source として必要なカラムを、`_construct_derived_columns` で pandas 派生する。`race_nkey`/`as_of_datetime`/`days_since_prev` の既存前例。
**When to use:** babacd（trackcd 分岐派生）・timediff（varchar→real parse）。両者とも `_DERIVED_HISTORY_COLUMN_NAMES` に追加し DB SELECT 対象から除外（BUG A fix 踏襲・同名衝突回避）。
**Example:**
```python
# Source: src/features/builder.py L120-205（既存 _construct_derived_columns・本フェーズ拡張）
_DERIVED_HISTORY_COLUMN_NAMES = frozenset({
    "race_nkey", "as_of_datetime", "days_since_prev",
    "timediff",   # NEW: varchar→real parse（sentinel 0000/9999→NaN）
    "babacd",     # NEW: trackcd 第1桁分岐派生（芝→sibaba・ダ→dirtbaba・不能→__MISSING__）
    "datakubun",  # 既存（D-05: 取り込まない・history SELECT にも含めない）
})

# _construct_derived_columns 内（with_history=True 时のみ構築）:
# timediff: raw varchar "[+-]NNN" / "0000" / "9999" → real（NNN/10 秒・sentinel は NaN）
if "timediff_raw" in result.columns and "timediff" not in result.columns:
    raw_td = result["timediff_raw"].astype(str)
    parsed = pd.to_numeric(raw_td, errors="coerce")  # 非数値(欠損等)→NaN
    sentinels = {"0000", "9999"}  # label_spec.timediff_sentinels 相当
    is_sentinel = raw_td.isin(sentinels)
    # "[+-]NNN" → 符号付き数値 / 10.0
    sign = raw_td.str[0]
    digits = raw_td.str[1:].astype("float")
    signed = digits.where(sign != "-", -digits)
    result["timediff"] = signed.where(~is_sentinel, np.nan) / 10.0
    result["timediff"] = result["timediff"].where(~is_sentinel, np.nan)

# babacd: trackcd 第1桁分岐
if all(c in result.columns for c in ("trackcd", "sibababacd", "dirtbabacd")) \
        and "babacd" not in result.columns:
    tc = result["trackcd"].astype(str).str.strip()
    lead = tc.str[0]
    baba = pd.Series(pd.NA, index=result.index, dtype=object)
    baba[lead.isin(["1", "2"])] = result.loc[lead.isin(["1", "2"]), "sibababacd"]
    baba[lead == "5"] = result.loc[lead == "5", "dirtbabacd"]
    # 欠損/曖昧/障害(5系でdirtbabacd='0'等)→__MISSING__ sentinel（D-13）
    baba = baba.where(baba.notna() & (baba != ""), "__MISSING__")
    result["babacd"] = baba
```

### Pattern 2: raw varchar pass-through + 消費側 cast（Phase 1 idiom）
**What:** normalized 層は raw の varchar をそのまま保持し、型変換は消費側（feature builder 等）で行う。Phase 1 の「全 varchar 明示キャスト」パターン。
**When to use:** timediff/baba3 の normalized への取り込み。DDL は varchar(4)/varchar(1)/varchar(2) で定義し、`_transform_*_df` では cast しない（消費側 builder で parse）。
**注意:** `timediff` は `_transform_uma_race_df` の real_cols(`futan`/`time`/`harontimel3` 等) には**追加しない**（varchar のまま保持・cast は builder 側）。これは既存 `kisyucode`/`jyocd` 等（varchar のまま）と一貫。

### Pattern 3: 3者 parity（registry↔rolling.py↔availability.py）
**What:** rolling 系統の定義が3箇所にあるため、`test_registry_rolling_systems_match_rolling_impl` が3者の一致を機械保証する。1箇所でも更新漏れがあれば RED。
**When to use:** timediff/babacd 再登録時。3箇所全てに同系統を追加（順序含め一致・`tuple` 比較あり）。

### Anti-Patterns to Avoid
- **timediff を `_transform_uma_race_df` で real cast しない:** varchar のまま normalized に保持し、消費側（builder）で parse + sentinel 処理。ETL 側で cast すると sentinel `0000`/`9999` の扱いが builder と分散し不整合の元。
- **babacd を DB 実カラムとして扱わない:** babacd は `_DERIVED_HISTORY_COLUMN_NAMES` に入れ、history SELECT には `babacd` を含めない（`sibababacd`/`dirtbabacd`/`trackcd` を SELECT して派生）。BUG A fix の構造踏襲。
- **trackcd 第1桁以外での芝/ダ判別:** 実データで第1桁 `1`/`2`=芝・`5`=ダート/障害と確定したため、この単純分岐のみ。`coursekubuncd` での判別は A-E のコース番号であり芝/ダ判別ではない（混同注意）。
- **babacd 派生で `__MISSING__` を忘れない:** trackcd 欠損/空/第1桁 `0`(未設定)/3,4(存在しない) の場合は `__MISSING__` sentinel（D-13）。silent NaN fill 禁止。
- **advisory hardening で silent empty を許容しない:** WR-02 の「チョークポイント検出」は `build_feature_matrix` で「空結果/全NaN」を単一検証点で fail-loud。`_fetch` 系は D-04通り log+空DF 返却のままだが、呼出側で必ず検知。

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| timediff parse | 正規表現自作 parser | `pd.to_numeric` + ベクトル化 `str[0]`/`str[1:]` | pandas 既存機能。`errors="coerce"` で非数値→NaN。sentinel 集合 `isin` で安全。 |

| rolling 集約 | 自作 loop | 既存 `build_rolling_features` pipeline | rolling.py の per-observation latest-K algorithm（obs_id group・strict `< cutoff`・D-13 sentinel）を再利用。新系統は `_ROLLING_SYSTEMS`/`_SYSTEM_SOURCE` 追加のみ。 |

## Common Pitfalls

### Pitfall 1: timediff sentinel `0000` の見落とし
**What goes wrong:** CONTEXT.md は sentinel を `9999` のみ記載。実データ検証で先頭`0`の376行が全て `0000` で kakuteijyuni='00'（取消/除外/中止）に完全一致することが判明。`0000` を sentinel 扱いしないと、取消馬行が timediff=0.0秒として rolling に混入する。
**Why it happens:** EveryDB2 04-UMA_RACE.md field 66 TimeDIFN は sentinel を `9999` のみ明記。`0000` は仕様書に明記されていないが、実データでは取消/除外/中止の代替 sentinel として使われている（`+000`=同タイム2着 とは別・kakuteijyuni='00' で識別）。
**How to avoid:** `label_spec.yaml` の `timediff_sentinels` に `"0000"`/`"0000.0"` を追加（既存 `"9999"`/`"9999.0"` と併せ）。builder 側の parse で `isin({"0000","9999"})` を sentinel 扱い。
**Warning signs:** rolling_timediff_mean_5 に `0.0` が異常に多く出現する・kakuteijyuni='00' の馬行が rolling に寄与する。

### Pitfall 2: babacd 派生で trackcd 第1桁 `5` の障害戦混入
**What goes wrong:** trackcd 第1桁 `5` はダート障害(trackcd 52・dirtbabacd active)と芝障害(trackcd 54/56・sibababacd active)が混在。単純に「`5`→dirtbabacd」とすると芝障害(trackcd 54/56)の babacd が `__MISSING__` になる（dirtbabacd='0' のため）。
**Why it happens:** 障害競走は芝とダートを跨ぐコース。trackcd だけでは芝/ダを完全判別できない。
**How to avoid:** 障害競走(syubetucd='18'/'19')は label_spec `model_ineligibility_syubetucd` で既に `is_model_eligible=False`（学習除外）。しかし **history 側（過去走）には障害戦が混入する**。babacd 派生では「`5`→dirtbabacd を見る・dirtbabacd='0'/欠損なら `__MISSING__`」とし、芝障害の馬場は取り込まない（保守的・D-13 sentinel 一貫）。あるいは「`5` かつ sibabababd≠'0' なら sibababacd」と第2条件を入れる。**推奨:** 第1桁 `1`/`2`→sibababacd・`5`→dirtbabacd（dirtbabacd='0'なら `__MISSING__`）の単純分岐（障害戦の馬場信号は Phase 1-A では副次的・GBDT が `__MISSING__` を処理）。
**Warning signs:** rolling_babacd が `__MISSING__` だらけになる・障害戦の過去走割合が異常に高い馬で babacd が欠損する。

### Pitfall 3: `_DERIVED_HISTORY_COLUMN_NAMES` と history SELECT の不整合
**What goes wrong:** timediff/babacd を history SELECT（`_HISTORY_DB_SELECT_COLUMNS`）に追加したまま `_DERIVED_HISTORY_COLUMN_NAMES` にも入れると、`_HISTORY_SELECT_COLUMNS`（= SELECT 名 − derived）から除外されるため、`_HISTORY_DB_SELECT_COLUMNS` に `ur.timediff AS timediff` があるのに `_HISTORY_SELECT_COLUMNS` に含まれない不整合が生じ、`assert TARGET_OBS_BANNED_COLUMNS.isdisjoint(_HISTORY_SELECT_COLUMNS)` 等の検査とズレる。
**Why it happens:** 既存 BUG A fix の構造（`timediff`/`babacd`/`datakubun` は DB 非実在扱い）を踏襲する際、実カラム timediff を SELECT に追加することを忘れる、または派生名と衝突させる。
**How to avoid:** timediff は raw 実カラムなので SELECT するが、**SELECT alias を `timediff_raw` にして `_HISTORY_SELECT_COLUMN_NAMES` は `timediff_raw`** とし、`_construct_derived_columns` で `timediff_raw`→`timediff`(real) を派生。babacd は SELECT せず（baba3 を SELECT して派生）。`_DERIVED_HISTORY_COLUMN_NAMES` には `timediff`/`babacd` を維持（これらは `_HISTORY_SELECT_COLUMNS` から除外される・derived 扱い）。これで BUG A fix 構造と整合。
**Warning signs:** `test_banned_columns_not_selected` が RED・`_HISTORY_SELECT_COLUMNS` に derived 名が混入。

### Pitfall 4: WR-03 vector化で SHA256 drift
**What goes wrong:** rolling.py L236-240 の `groupby().apply(lambda s: int(s.notna().sum()))` を vector化する際、count 計算結果が微妙に変わると（例: NaN 扱いの差で count が ±1）、feature_matrix の値が変わり SC#3 byte-reproducibility が破れる。
**Why it happens:** pandas 3.x で `apply` と `transform`/vector化で浮動小数点演算順序が変わるとMean等の値が bit-level で変わる可能性。count は整数だが mean/latest/sd の計算順序も関連。
**How to avoid:** vector化は **count のみ**（`groupby().apply(lambda s: s.notna().sum())` → `groupby()._sys_value` の `notna().groupby().sum()`）。mean/latest/sd は既存 `groupby().mean()`/`.first()`/`.std()` を維持（これらは既に vector化されている・SHA256 維持）。WR-03 の対象は L236-240 の count 計算のみ。変更後、既存 snapshot と SHA256 比較で一致を assert（ただし本フェーズは feature_count 18→24 で snapshot 自体が変わるため、WR-03 の SHA256 維持検証は「vector化前後の同一入力で同一 rolling_df」の unit test で証明）。
**Warning signs:** `test_byte_reproducible_by_hash` RED・count 値の ±1 ズレ。

### Pitfall 5: CR-01新 persist→manifest 順序で partial-failure 状態
**What goes wrong:** persist_category_maps を manifest 書出より先に実行する際、persist が成功しても manifest 書出が失敗すると、category_map artifact だけが残り manifest が無い不整合状態になる。
**Why it happens:** 順序を変えるだけでは partial-failure を完全に防げない。トランザクション性が必要だが Parquet/YAML/JSON は別ファイル。
**How to avoid:** (1) persist → (2) `assert category_map_path.exists()` → (3) manifest 書出（SHA256 含む）。manifest の `category_map_artifact` を repo-root 相対（`snapshots/category_map_*.json`）で正規化し CWD 依存を除去。persist 失敗時は例外で manifest 書出前に停止（manifest が無い＝未完成状態・明確）。
**Warning signs:** manifest が無いのに category_map artifact がある・`category_map_artifact` が CWD 依存で run 毎に変わる。

### timediff parse（実データ検証済み仕様）
```python
# Source: 実データ検証（2026-06-19・readonly ロール）+ EveryDB2 04-UMA_RACE.md field 66
# timediff varchar(4): 全554,267行 length=4・NULL=0
# 符号分布: '+'509,756 / '-'39,629 / '9'(=9999 sentinel)4,506 / '0'(=0000 sentinel)376
# 1着馬(kakuteijyuni='01'): 全て '-' 符号・MIN='-000'(同タイム2着)・MAX='-056'(5.6秒差)
# 2着以下: 全て '+' 符号
# sentinel '0000'(376件)・'9999'(4,506件) は kakuteijyuni='00'(取消/除外/中止)に完全一致
# label_spec.yaml timediff_sentinels に "0000"/"0000.0" を追加要（現在 "9999"/"9999.0" のみ）

import numpy as np
import pandas as pd

def parse_timediff_series(raw: pd.Series) -> pd.Series:
    """timediff varchar '[+-]NNN'/'0000'/'9999' → real（NNN/10秒・sentinel は NaN）。

    label_spec.se_marker_canonicalization.timediff_sentinels と同一 sentinel 集合。
    """
    s = raw.astype(str).str.strip()
    sentinels = {"0000", "9999"}  # label_spec と同期
    is_sentinel = s.isin(sentinels)
    # 符号 + 3桁数字
    sign = s.str[0]
    digits = pd.to_numeric(s.str.slice(1, 4), errors="coerce")  # 非数値→NaN
    signed = digits.where(sign != "-", -digits)
    value_seconds = signed / 10.0  # NNN/10 秒
    return value_seconds.where(~is_sentinel, np.nan)
```

### babacd trackcd 分岐派生（実データ検証済み仕様）
```python
# Source: 実データ検証（2026-06-19）+ EveryDB2 03-RACE.md field 36/66/67
# trackcd 実分布（第1桁）: '1'/'2'=芝(sibababacd active・dirtbabacd='0')
#                         '5'=ダート/障害混合(trackcd 52=dirt active, 54/56=siba active)
# sibababacd 値域: '0'(19,151)/'1'(15,958)/'2'(3,055)/'3'(1,169)/'4'(260)
# dirtbabacd 値域: '0'(20,022)/'1'(12,041)/'2'(4,076)/'3'(2,249)/'4'(1,205)
# CODE.md §2010 は不完全('0'のみ)→実データ値 0-4 を生序数扱い(D-02・jyocd 前例)
# 障害競走(syubetucd='18'/'19')は model_ineligibility で学習除外済みだが history に混入

def derive_babacd(trackcd: pd.Series, sibababacd: pd.Series, dirtbabacd: pd.Series) -> pd.Series:
    """trackcd 第1桁で芝/ダを判別し babacd を派生（不能時 __MISSING__ sentinel）。

    D-13 silent fallback 禁止・days_since_prev/race_nkey と同枠の派生 history カラム。
    """
    from src.utils.category_map import MISSING
    tc = trackcd.astype(str).str.strip()
    lead = tc.str[0]
    baba = pd.Series(pd.NA, index=trackcd.index, dtype=object)
    # 芝: trackcd 第1桁 '1'/'2'
    siba_mask = lead.isin(["1", "2"])
    baba[siba_mask] = sibababacd[siba_mask]
    # ダート/障害: trackcd 第1桁 '5' → dirtbabacd を見る('0'/欠損なら __MISSING__)
    dirt_mask = (lead == "5")
    dirt_val = dirtbabacd[dirt_mask]
    baba[dirt_mask] = dirt_val.where(dirt_val.notna() & (dirt_val != "0"), MISSING)
    # 欠損/曖昧/未設定('0'・第1桁 '0'/'3'/'4')→ __MISSING__
    baba = baba.where(baba.notna() & (baba != "") & (baba != "0"), MISSING)
    return baba
```

   - What we know: `0000` は sentinel（kakuteijyuni='00' と完全一致）。現在 label_spec は `9999`/`9999.0` のみ。
   - What's unclear: label_spec.timediff_sentinels は se_marker canonicalization 用（label 層）。rolling 層の timediff parse は builder 別物。両方に追加するか、builder 側ハードコードか。
   - Recommendation: **builder 側に定数として持つ**（`_TIMEDIFF_SENTINELS = frozenset({"0000", "9999"})`）。label_spec.timediff_sentinels は label 層（se_marker）用のため、rolling 層とは関心事が異なる。ただし将来の単一の真の源のため、`label_spec.yaml` に `rolling_timediff_sentinels` を追加する選択肢もある（planner 判断）。実害はどちらでもない。
   - RESOLVED: **両方に "0000" を含める**（planner 採択・Plan 03 Task1/Task3 で実装）。builder 側に `frozenset({"0000", "9999"})` 定数を持ち（rolling 層の timediff parse 用・Task1）・`label_spec.yaml` の `timediff_sentinels` にも `"0000"`/`"0000.0"` を追加（label 層 se_marker canonicalization 用・Task3）。両層で sentinel 集合が "0000"/"9999" で整合する。

2. **babacd 派生で trackcd 第1桁 `5` かつ dirtbabacd='0' の芝障害(trackcd 54/56)の扱い**
   - What we know: trackcd 54/56 は sibababacd active・dirtbabacd='0'。単純「`5`→dirtbabacd」だと芝障害は `__MISSING__`。
   - What's unclear: 芝障害の馬場信号を取り込むか（第2条件「`5` かつ sibababacd≠'0' なら sibababacd」）。障害戦は model_ineligibility だが history に混入。
   - Recommendation: **単純分岐（`5`→dirtbabacd・'0'なら `__MISSING__`）を採用**。理由: (a) 障害戦の過去走を持つ馬は少数・GBDT が `__MISSING__` を処理、(b) 芝障害と芝平地の馬場状態コード体系が同一とは限らない（障害コースは独自）、(c) 保守的・D-13 一貫。Phase 4 評価後に改善余地（D-08 deferred）。
   - RESOLVED: **単純分岐（第1桁 1/2→sibababacd・5→dirtbabacd・'0'/欠損なら `__MISSING__`）を採用**（planner 採択・Plan 03 Task1 で実装）。芝障害(trackcd 54/56)の babacd は `__MISSING__` sentinel となる（保守的・D-13 一貫・GBDT が `__MISSING__` を処理）。
