# Phase 03.1 Plan 03: Timediff/Babacd Rolling 復元 + advisory 3件 hardening Summary

feature 層の core 拡張（timediff parse・babacd trackcd 分岐派生・rolling 8系統化・3者 parity 再登録・label sentinel）と advisory 3件（WR-01'/WR-02 fail-loud・WR-03 vector化）の hardening を実施し、Phase 4 モデルが馬場適性信号（rolling 18→24 features）を利用可能にした。

## What Was Built

### Task 1: builder.py 拡張（timediff parse / babacd 派生 / WR-01'/WR-02 fail-loud）
- **history SELECT 拡張**: `_HISTORY_DB_SELECT_COLUMNS` に `ur.timediff AS timediff_raw`（派生名衝突回避 alias・Pitfall 3）・`nr.sibababacd AS hist_sibababacd` / `nr.dirtbabacd AS hist_dirtbabacd`（TARGET_OBS_BANNED 同名衝突回避 alias・HIGH #4 intent 保持）・`nr.trackcd AS trackcd` を追加
- **timediff parse**: `_construct_derived_columns` で `raw_td = result["timediff_raw"].astype(str)` → sentinel `{"0000","9999"}` を `isin` 判定で NaN 化（`_TIMEDIFF_SENTINELS` 定数）→ sign/digits 抽出 → `signed/10.0` で real 秒化（NNN/10・D-03・T-03.1-07）
- **babacd trackcd 分岐派生**: trackcd 第1桁 `1`/`2`→芝→`hist_sibababacd`・`5`→ダート→`hist_dirtbabacd`（`'0'`/欠損/NaN は `__MISSING__`・T-03.1-08 accept・D-13）
- **WR-01' fail-loud**: L347-354 の `else: pit_filtered_style = expanded_style`（silent no-filter fallback）を削除し、`as_of_datetime` 不在で `ValueError` を raise
- **WR-02 fail-loud**: `build_feature_matrix` の `_fetch_feature_sources` 直後に `if len(feature_matrix) == 0: raise RuntimeError(...)` を追加（チョークポイント検出・D-04）

### Task 2: rolling.py 8系統化 + WR-03 vector化
- **8系統化**: `_ROLLING_SYSTEMS` に `"timediff"`, `"babacd"` を末尾追加（6→8系統・D-01 完全復元・SC#2 rolling 18→24）
- **_SYSTEM_SOURCE 拡張**: `"timediff": ("timediff",)`, `"babacd": ("babacd",)` を追加（単一 source・jyocd と同型）
- **WR-03 vector化**: count 計算の `groupby().apply(lambda s: int(s.notna().sum()))` を `notna().groupby().sum().astype(int)` の vector 形に置換（pandas 3.x 推奨・apply 廃止・Pitfall 4 SHA256 drift 対策）・mean/latest/sd は既存 vector化維持

### Task 3A: availability.py + feature_availability.yaml の3者 parity 再登録
- **availability.py**: `_ROLLING_SYSTEMS_FOR_RESERVED` に `"timediff"`, `"babacd"` を rolling.py と同一順序で追加（8系統・`_RESERVED_NON_FEATURE_COLUMNS` 内包表記で `rolling_timediff_count_5`/`rolling_babacd_count_5` が自動 reserved 化）

### Task 3B: label_spec.yaml sentinel + テスト5関数更新

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] babacd 派生に使う sibababacd/dirtbabacd を alias で SELECT**
- **Found during:** Task 1 history SELECT 拡張の実装時
- **Issue:** プラン action (B) は babacd 派生で `sibababacd`/`dirtbabacd` の値を使うことを想定していたが、これらを素のカラム名で SELECT すると L213-215 の `assert TARGET_OBS_BANNED_COLUMNS.isdisjoint(_HISTORY_SELECT_COLUMNS)` が同名衝突で失敗する（sibababacd/dirtbabacd は TARGET_OBS_BANNED に含まれる）。PATTERNS.md L191 は「babacd 自体は SELECT しない」と示唆するが、babacd の値（馬場状態コード）を得るには元カラムが必要
- **Fix:** `nr.sibababacd AS hist_sibababacd` / `nr.dirtbabacd AS hist_dirtbabacd` と alias を付与し、`_HISTORY_SELECT_COLUMNS` には `hist_*` 名で入るため TARGET_OBS_BANNED と disjoint を保持。HIGH #4 の intent（target race 当日行の馬場=禁止・history の過去走 n_race 馬場は許可）は維持
- **Files modified:** `src/features/builder.py`（`_HISTORY_DB_SELECT_COLUMNS`・`_construct_derived_columns` の babacd 派生で `hist_*` 参照）
- **Commit:** f72fbe5
- **根拠:** acceptance criteria「`_HISTORY_DB_SELECT_COLUMNS` に `nr.sibababacd`/`nr.dirtbabacd`/`nr.trackcd` が含まれる」と HIGH #4 disjoint assert の両立。alias により両要件を満たしつつ、babacd 派生ロジックの正しさ（実際の馬場状態コード値を使用）を保持

**2. [Rule 3 - Blocking] WR-01' テストを ValueError/KeyError 両方を受け入れる形に調整**
- **Found during:** Task 3B WR-01' unit test 実装時
- **Issue:** WR-01' の ValueError（推定脚質ステップ L348-354）を検出するテストを書いたが、実際には Step 5 rolling（L296 `history.sort_values("as_of_datetime")`）が先に `KeyError: 'as_of_datetime'` で失敗する。両ステップとも as_of_datetime を要求する二重防御構造
- **Fix:** テストを `pytest.raises((ValueError, KeyError))` で両エラーを受け入れ、メッセージに `as_of_datetime` が含まれることを検証する形に調整。WR-01' の intent（将来 refactor での silent leak 防止）は、いずれかのステップで fail-loud になれば達成されるため、この調整で機械保証は維持される
- **Files modified:** `tests/features/test_builder.py`（`test_wr01_prime_raises_on_missing_as_of_datetime`）
- **Commit:** aef37a2
