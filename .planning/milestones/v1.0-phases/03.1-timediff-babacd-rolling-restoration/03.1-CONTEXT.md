### ⚠ 実DB検証による前提修正（重要・researcher/planner 必読）

roadmap SC#1/SC#2 は「`timediff`/`babacd` を normalized に取り込む」と記載していたが、**live-DB 検証で `babacd` が実カラム非存在と判明**。これが本フェーズの主作業を「機械的カラム取り込み」から「設計特徴量の派生実装」に変える：

| カラム | live-DB 実態 | 復元方法 |
|---|---|---|
| `timediff` | ✅ `public.n_uma_race` に実在（varchar・`+000`〜`9999`・sentinel `9999`=8,554件≈1.0%・NULL無し） | ETL SELECT 1カラム追加（機械的） |
| `babacd` | ❌ **実在しない**（n_uma_race/n_race のいずれにも babacd 無）。馬場系は `n_race` の `sibababacd`/`dirtbabacd`/`trackcd` のみ | `normalized.n_race` 拡張 + history で `trackcd` 分岐派生（設計+実装） |
| `datakubun` | raw に実在・**normalized 非存在**。`normalized.n_uma_race` は既に 554,267 行に dedupe 済み（version選別=ETL側完結） | **取り込まない**（復元に不要・D-05） |

元 Phase 3 設計者は babacd を「normalized にあるべきカラム」と想定していたが（STATE.md の D-13 sentinel fallback ノートも示唆）、実在しなかったのが 03-05 全NaN breach の真因。実値が流れる本フェーズが、この設計の初めての実検証になる。

## Implementation Decisions

### babacd 復元方針（最大の決定）
- **D-01: babacd は完全復元（24 feature・8系統×3軸）** — `normalized.n_race` に `sibababacd`(芝馬場)/`dirtbabacd`(ダート馬場)/`trackcd`(芝ダ判別) の3カラムを追加し、history builder で各過去レース行の `trackcd` を見て対応する馬場カラムを選択し babacd を派生する。babacd は `days_since_prev`/`race_nkey` と同じ「派生 history カラム」として扱う。SC#2（rolling 18→24）を維持。ユーザー選択: A（完全復元）。馬場適性信号を Phase 4 モデルに入れる意図。
- **D-02: babacd 集約 = mean/latest/sd の3軸・生の序数コードを数値扱い（特別リマップなし）** — `jyocd`（競馬場コード=2桁序数コード）の既存 rolling 前例と一貫。GBDT は序数値を自然に処理するため mean/sd も意味を持つ。`babacd` code→序数の順序付けは researcher が `docs/everydb2/CODE.md#2010`（馬場状態コード・スクレイピング不完全・要補完）で検証する。特別リマップは将来の改善余地として D-08 に defer。

### timediff 取り込み
- **D-03: timediff は標準（parse + sentinel除外）** — 符号付き文字列 `[+-]NNN`（NNN=3桁・値=`NNN/10`秒・1着馬は負値・2着以下は正值）を real に parse。sentinel `9999`（取消/除外/中止）は rolling 集約から除外→NaN。取消/除外/中止の過去走は timediff に寄与せず、既存 rolling の `__MISSING__`/count 扱いと一貫。parse 書式の詳細（先頭`0`の1,028行 edge case 等）は researcher が実データで確定。ユーザー選択: 標準。

### advisory hardening
- **D-04: advisory 4件の hardening は todo 処方箝通り・Claude/計画者に委ねる** — 各 advisory の file:line + 修正方針 + テスト条件は `.planning/todos/pending/phase3-advisory-hardening.md` に固定済み。WR-02 の fail-loud 実装スタイルは「チョークポイント検出」を採用（`_fetch` は log+空DF返却のまま、`build_feature_matrix` が「空結果/全NaN」を単一検証点で fail-loud → DB例外・0行結果・全NaN の全 silent-empty 系を一括捕捉・メッセージ明確）。ユーザー選択: 任せる。

### スコープ厳格
- **D-05: datakubun は normalized に取り込まない** — `normalized.n_uma_race` は既に 554,267 行に dedupe 済み（version選別はETL側完結）のため、timediff 取得・babacd 派生のいずれにも不要。roadmap SC#1（timediff/babacd のみ）を厳格に維持。将来の細かい history フィルタ用途（障害戦除外等）は D-08 に defer。ユーザー選択: 取り込まない。

### Reusable Assets
- **`src/features/rolling.py` の rolling framework** — `_ROLLING_SYSTEMS`/`_SYSTEM_SOURCE` に `timediff`/`babacd` を追加するだけで、per-observation latest-K algorithm（strict `< cutoff`・obs_id=(race_nkey,kettonum) group）が mean/latest/sd/count を自動生成。timediff は source カラムそのまま・babacd は派生 history カラムを source に指定

## Specific Ideas

- **timediff parse 仕様（実DB検証確定）** — `[+-]NNN`（NNN=3桁・値=`NNN/10`秒）。1着馬は負値（`-000`〜`-019`...=勝差・`-019`=1.9秒差で勝利）・2着以下は正值。符号分布: `+`800,330件 / `-`71,290件 / 先頭`0`1,028件（edge case・researcher検証）。sentinel `9999`=取消/除外/中止=8,554件≈1.0%
- **babacd 派生設計** — 過去レース行ごと: `trackcd` が芝 → `sibababacd`、ダート → `dirtbabacd`。`normalized.n_race` に3カラム追加（`sibababacd`/`dirtbabacd`/`trackcd`・全 varchar）→ history builder で `race_nkey` 結合 + `trackcd` 分岐で派生。`normalized.n_race` は1行/レース(dedupe済み・Phase 1 39,593レース)なので結合は clean
- **normalized.n_uma_race の dedupe 事実** — 554,267行（=label行数）・version選別はETL側完結（`datakubun` 非 SELECT でも最終版1行/(レース,馬)）。timediff はこの dedupe 済みデータから素直に取得
- **babacd 復元は本来の主作業** — 機械的カラム取り込み(timediff)ではなく設計特徴量の派生実装。`normalized.n_race` スキーマ拡張（target_obs_banned だった馬場カラムを history 用に取り込む・TARGET race には絶対使わないよう PIT/history 区分で安全管理）+ history builder の `trackcd` 分岐ロジック
- **feature_count 18→24** — manifest/§12.4 metadata に反映。新 `feature_snapshot_id`/`dataset_version`。live snapshot rebuild で byte-repro + registry↔Parquet parity 再検証
