# v1.1 ドメイン分析（外部 2AI リサーチ統合）

**Defined:** 2026-06-25
**Source:** ChatGPT / Codex に同一プロンプトで調査依頼（プロンプトは `v1.1` マイルストーン定義時に設計・ユーザーが別セッションで実施）
**Purpose:** 回収率0.65天井（debug `fukusho-recovery-070` ROOT CAUSE）へ core value（odds-free）を維持したまま正統に対応するため、競馬ドメイン知識・ベッティング市場経済学の学術研究・実践 ML プロジェクトから「何が足りていないか」を分析。

---

## 0. 結論（2AI 完全一致）

**現状の odds-free・62特徴量モデルを閾値調整だけで回収率1.0超へ持っていくのはほぼ無理。** モデル精度が低いのではなく、**「市場が低評価している馬」を選ぶ局面（投票層）だけでモデル誤差が激しく表面化**しているため（AUC/Brier/LogLoss は良好なのに EV 選定層で高オッズ馬の確率を4倍前後過大評価）。

**core value を維持したままの勝ち筋（2AI 一致）:**
1. 能力特徴量を「着順中心」から「**速度（スピード指数）・相手強度・展開・適性・状態**」へ拡張
2. 独立二値分類ではなく**レース内相対確率モデル**へ近づける
3. EV 判定に点推定 `p` でなく **`p` の下側信頼限界（`p_lower`）** で買う／見送る

**正直な結論:** 複勝・odds-free・前日cutoff だけで長期回収率1.0超を安定達成する可能性は低い（市場は時間経過で情報集約が進む）。現実シナリオ **0.78-0.92**・悲観 **0.65-0.80 天井**・楽観 0.90-1.05。2AI とも「現実シナリオが最も可能性高い」と評価。

## 1. 前提の修正（2AI 一致・重要）

**「`p` とオッズの独立性」は数学的に強すぎる。** 優秀な能力モデルほど `p` とオッズは負相関し、独立にはならない。正しい必要条件は：

```
必要条件: モデル p が、オッズに繰込済みの平均的市場評価を超えて、
         条件付きで正しく校正（calibrated）されていること。
特に重要: odds帯・EV帯・モデル市場乖離帯で、p が過大評価されていないこと。
```

→ **core value 再定式化:** 「オッズを `p` モデルの特徴量に入れない。ただし `p` の校正・誤差・EV成績はオッズ帯別に診断し、EV 判定では市場オッズを使う。」（要件§2.2/§9.3 整合・PROJECT.md Key Decisions に明文化済）

## 2. 市場効率性と複勝 pool の搾取可能性

- **FLB（favorite-longshot bias）**: 長期的に人気馬の期待損失は小さく、穴馬ほど過大評価で期待損失大。現在の「中高オッズ帯過大p で高オッズ馬を拾って赤字」は FLB に乗っている状態と整合。
- **Dr. Z system（Hausch/Ziemba/Rubinstein 1981）**: place/show pool の非効率・プール間非整合を利用する古典。**純粋能力モデルでなく市場プール構造の歪み利用** → core value（odds-free `p` モデル）とは直接相性不良。使うなら `p` モデルでなく EV/見送り/最終払戻補正レイヤー。
- **複勝 edge は出にくい**: 「人気薄全般」でなく「市場が嫌うが能力・展開・適性上は3着以内確率が落ちていない馬」でのみ出やすい。JRA 複勝払戻率 80%（控除率20%）→ 黒字には控除後市場価格を25-30%上回る確率推定が必要。

## 3. 現システムのギャップ（2AI 一致）

**強み:** リーク防止・再現性の基礎は良好（PIT特徴量・race_id時系列分割・baseline 分離・BL-3 を不公平比較と扱う設計）。

**最大の不足: 能力特徴量が「市場と戦える密度」でない（「結果の要約」であって「能力値そのもの」でない）。**

| 不足領域 | 現状 | なぜ致命的 |
|---------|------|-----------|
| **スピード指数** | 生タイム・着差・上がりはあるが馬場/距離/クラス/トラック補正後の指数がない | 着順より能力を直接測る主軸 |
| **相手強度補正** | 過去レースの相手が強かったか弱かったか未反映 | 低着順でも強い相手なら価値 |
| **ペース・展開** | 通過順ベースの脚質のみ・レース全体の前傾/後傾・逃げ馬数・先行圧力が弱い | 複勝は展開不利で4-6着に沈む馬の判定が重要 |
| **距離・馬場・コース適性** | sire/bms ID はあるが適性マッピング未整備 | 人気が嫌う条件替わりの拾い上げ |
| **状態面** | 馬体重・調教・休み明け内容・叩き2走目が弱い | 市場が最も重視する直近状態に対抗不能 |
| **レース内相対確率** | 各馬独立の二値分類に近い | 複勝対象数の制約（8頭以上3頭/5-7頭2頭）と相手構成を表現できない |

**根本ギャップ:** 全体では勝つのに「市場と大きく意見が割れた馬だけ買っている」→ その乖離の多くがモデルの見落としでなく市場の正しさだった。閾値を締めると「より強く市場に逆らった馬」に集中し過大p が濃縮されるため閾値調整では解消しない。

## 5. ML 設計の改善方針（2AI 一致）

**二値分類の限界:** 複勝は各馬独立事象でない（払戻対象数の構造制約）。推奨段階:
1. odds-free 能力スコア `s_i` を作る
2. 同一 race_id 内で相対化（rank / gap_to_top / gap_to_median / field_strength）
3. `p_fukusho_hit` を校正（`sum(p)` が払戻対象数に近づくよう制約/補正）
4. `p_lower × odds_lower` で EV 判定

**現要件の `sum(p)` チェックは「診断」でなく「補正対象」に格上げすべき。**

**focal loss は慎重:** 難例へ学習集中で確率校正が悪化する恐れ。`p` 校正を守るなら通常 logloss/Brier 最適化 + false-edge 特徴量追加 + `p_lower` 選定が良い。

**profit 最適化は `p` モデルでなく selector:** `p` モデルは odds-free のまま。層分離:
| 層 | 使用情報 | 目的 |
|----|---------|------|
| `p` モデル | odds-free 特徴量のみ | 真の複勝払戻対象確率 |
| calibration 層 | odds-free・時系列 valid のみ | `p` の過信補正 |
| EV/selector 層 | `p`・`p_lower`・odds・snapshot policy | 買う/見送る |
| evaluation 層 | odds帯・人気帯・利益 | 市場に勝てるか診断 |

## 6. 評価指標の拡張（2AI 一致）

現要件（Calibration/人気帯/オッズ帯/競馬場/頭数/年別）は良い方向。追加すべき:

| 追加評価 | 目的 |
|---------|------|
| selected-only calibration | 実際に買う馬だけの `p` が正しいか |
| EV decile 別実現 ROI | EV が高いほど本当に儲かるか |
| model-market disagreement 別 ROI | 市場と大きく割れたときに勝てるか |
| odds snapshot→final payout slippage | 10分前/30分前オッズが実払戻をどれだけ過大評価するか |
| `p_lower` EV | 点推定でなく下側確率でも EV>1 か |
| false-edge analysis | 高p・高odds・不的中馬の共通原因特定 |
| **falsification test（Codex）** | `logit(outcome) ~ logit(market) + logit(model)` を時系列 out-of-sample で測り model 係数が利益に残るか。残らなければ「core value 維持での黒字化は棄却」。回収率0.65 が特徴量不足か構造的限界かの鑑別。**次マイルストーンに必須組み込み。** |

Brier 全体平均は高EV選定層の失敗を隠す（確率精度は測るが希少/意思決定領域の利益性は測らない）。
